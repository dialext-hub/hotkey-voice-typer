# Дизайн: стриминг как отдельная toggle-клавиша (редизайн)

**Дата:** 2026-06-30
**Статус:** Утверждён, готов к плану реализации
**Файл:** `voice_typer.py`
**Заменяет интерактив из:** `2026-06-26-streaming-chunk-transcription-design.md`

## Проблема

Первая версия стриминга (печать чанков во время удержания F9) на тесте 2026-06-30
сломала текст: чанк печатается, пока пользователь ещё физически держит push-to-talk
клавишу, и впрыск символов (`keyboard.write`) сталкивается с удерживаемой клавишей
и её автоповтором в очереди ввода Windows → начало слов съедается/утраивается
(«ну проверяем новый режим» → «ууу оверяем ввый еежим»). Это архитектурный тупик:
**впрыскивать текст, пока зажата активационная клавиша, нельзя.**

Дополнительно всплыли: фрагментация на коротких паузах (0.8 c режет внутри фраз)
и краш трея `OSError [WinError 1402]` при обновлении иконки из нескольких потоков.

## Цель

Переработать стриминг в **отдельную toggle-клавишу без удержания**: нажал — старт
сессии, чанки печатаются по паузам, нажал ещё раз — стоп. Клавиша не зажата во
время печати → коллизии нет. Фича экспериментальная, по умолчанию выключена,
включается одной строкой в конфиге. Обычный push-to-talk на F9 не меняется.

## Решение

### Конфиг

```json
{
  "streaming_hotkey": "f10",   // присутствует и непусто → стриминг включён на этой клавише; нет/пусто (дефолт) → выключено
  "pause_threshold": 1.0,      // дефолт поднят с 0.8 (0.8 дробил внутри фраз)
  "max_chunk_seconds": 15,
  "silence_threshold": 0.01
}
```

- `streaming_hotkey` парсится тем же `_parse_hotkey`, что и основной `hotkey`
  (поддерживает модификаторы: `"ctrl+f10"`).
- Пустая строка или отсутствие поля → стриминг выключен (дефолт).
- Старое поле `streaming: true` и путь «стриминг на удержании F9» **удаляются**.
- `pause_threshold` / `max_chunk_seconds` / `silence_threshold` — опциональны,
  читаются свежими при старте сессии (toggle-on); дефолт `pause_threshold` = 1.0.

### Интерактив (toggle, без удержания)

- Нажатие `streaming_hotkey` когда система простаивает → **старт** стриминг-сессии:
  `recorder.start_streaming(...)`, трей = `recording`. Чанки печатаются по паузам,
  пока клавиша НЕ зажата → впрыск проходит чисто.
- Повторное нажатие `streaming_hotkey` во время сессии → **стоп**: `recorder.stop()`,
  `recorder.flush_final()`, трей = `processing`; фоновый `_drain` ждёт `wait_idle()`
  и ставит `idle`.
- **Защита от автоповтора:** toggle срабатывает на первом `KEY_DOWN`; пока клавиша
  физически удерживается, повторные `down` игнорируются. Флаг `_stream_key_held`
  ставится на `KEY_DOWN`, сбрасывается на `KEY_UP`.

### Взаимное исключение

Одна запись за раз:
- идёт стриминг-сессия → `KEY_DOWN` основного `hotkey` игнорируется;
- идёт обычная push-to-talk запись (`is_recording` установлен) → нажатие
  `streaming_hotkey` игнорируется.

F9 push-to-talk (зажал → говори → отпусти → одна расшифровка) не меняется.

### Чистая функция решения

Логику `on_key_event` (теперь обрабатывает обе клавиши + toggle + взаимное
исключение) выносим в чистую тестируемую функцию:

```
decide_key_action(event_type, key_name, modifiers_ok, state) -> action
```

- `state` — снимок: `ptt_recording: bool`, `stream_active: bool`,
  `stream_key_held: bool`, плюс какие имена клавиш — основная и стриминговая.
- `action` ∈ `{"start_ptt", "stop_ptt", "start_stream", "stop_stream", "ignore"}`.
- Без побочных эффектов и без обращения к клавиатуре/звуку → юнит-тестируется
  последовательностями событий (включая автоповтор). `on_key_event` остаётся
  тонкой обёрткой: собрать state → вызвать `decide_key_action` → выполнить action.

### Компоненты

- **Без изменений (переиспользуем, уже на ветке):** `ChunkDetector`,
  `ChunkTranscriber`, `AudioRecorder.start_streaming/flush_final/_emit_chunk`,
  `handle_chunk`.
- **`TrayIcon.set_state` — фикс краша:** оборачиваем тело в `threading.Lock`
  (`_state_lock`), чтобы параллельные обновления иконки (воркер / `_drain` /
  таймер ошибки) не пересекались. Это убирает `WinError 1402` (две нити
  одновременно дёргают `_update_icon`/`DestroyIcon`).
- **`main()`:** парсим обе клавиши в `cfg` (`trigger_key`/`modifiers` +
  `stream_trigger`/`stream_modifiers`); состояния `is_recording` (есть),
  `stream_active` (новый `threading.Event`), `_stream_key_held` (новый bool);
  `on_key_event` — тонкая обёртка над `decide_key_action`; `reload_config`
  перечитывает обе клавиши.

### Поток данных (стриминг-сессия)

```
streaming_hotkey DOWN (простаивает) → start_streaming , stream_active.set() , tray=recording
   ... во время сессии (клавиша НЕ зажата): паузы → чанки в очередь → воркер печатает по порядку
streaming_hotkey DOWN (сессия активна) → stop() , flush_final() , stream_active.clear() , tray=processing
   очередь опустела → tray=idle
```

### Краевые случаи

- Автоповтор `streaming_hotkey` (зажал) → один toggle на физическое нажатие.
- `reload_config` со сменой `streaming_hotkey` → обновляем `cfg`, как для основной.
- Пустой/отсутствующий `streaming_hotkey` → `decide_key_action` никогда не вернёт
  `start_stream`/`stop_stream` (стриминговое имя клавиши = None).
- Автозамены через границу чанка не срабатывают — осознанное ограничение.

## Тестирование

- **Юнит `decide_key_action`:**
  - tap streaming-key из простоя → `start_stream`; tap снова → `stop_stream`.
  - автоповтор `down` при `stream_key_held=True` → `ignore` (один старт на нажатие).
  - `down` основной клавиши при `stream_active=True` → `ignore` (взаимное исключение).
  - `down` streaming-key при `ptt_recording=True` → `ignore`.
  - `streaming_hotkey=None` → стриминговые ветки недостижимы.
  - основная push-to-talk (`down`/`up`) даёт `start_ptt`/`stop_ptt` как раньше.
- **`ChunkDetector` / `ChunkTranscriber`** — существующие 11 тестов остаются зелёными.
- **Ручная проверка:** F10 → речь с паузами → текст печатается чисто (без съеденных
  букв) → F10 стоп; долгая речь без пауз → флаш по `max_chunk_seconds`; F9
  push-to-talk работает по-старому; быстрый автоповтор F10 не дёргает старт/стоп.
